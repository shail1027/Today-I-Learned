# 웹 서버와 통신 기초

## 서버란 무엇인가?

**서버(Server)** 란 특정 기능이나 데이터를 요청하는 **클라이언트(Client)** 에게 응답하는 컴퓨터 또는 프로그램이다. "서버"는 고성능 하드웨어만을 가리키는 것이 아니라, **역할**을 나타내는 개념이다. 내 노트북도 서버 프로그램을 실행하면 서버가 된다.

```
[ 클라이언트 ]  ──── 요청(Request) ────>  [ 서버 ]
                <─── 응답(Response) ────
```

우리가 브라우저에서 유튜브를 열 때, 브라우저가 클라이언트가 되고 유튜브 서버에 "이 영상 목록을 줘"라고 요청하는 것이다.

## 웹 서버 vs WAS

서버는 하는 일에 따라 구분된다.

### 웹 서버 (Web Server)

**정적인 파일**을 그대로 돌려주는 서버다. HTML, CSS, JS, 이미지 파일처럼 누가 요청해도 내용이 변하지 않는 파일을 빠르게 전달하는 역할을 한다. 대표적으로 **Nginx**, **Apache**가 있다.

### WAS — 웹 애플리케이션 서버 (Web Application Server)

**동적인 콘텐츠**를 처리하는 서버다. 로그인 상태에 따라 다른 화면을 보여주거나, DB에서 데이터를 조회해 가공하는 등 **비즈니스 로직**을 실행한다. Spring Boot, Django, FastAPI, Node.js(Express) 등이 여기에 해당한다.

```
브라우저
  │
  ▼
[ Nginx - 웹 서버 ]   ← 정적 파일(.html, .css, .jpg)은 여기서 바로 응답
  │
  │  동적 요청(/api/...)만 뒤로 넘김
  ▼
[ Spring/Django/FastAPI - WAS ]   ← 비즈니스 로직 처리
  │
  ▼
[ DB (PostgreSQL, MySQL...) ]
```

실무에서는 웹 서버(Nginx)를 앞에 두고, 동적 요청만 WAS로 **전달(프록시)** 하는 구성을 많이 쓴다. Nginx가 정적 파일은 직접 처리하고, SSL 처리, 요청 분산 등도 맡아 WAS의 부담을 줄인다.

## 인터넷에서 서버를 찾는 방법 — IP와 DNS

### IP 주소

인터넷에 연결된 모든 컴퓨터는 **IP 주소(Internet Protocol Address)** 를 가진다. 집 주소처럼, 네트워크에서 특정 컴퓨터를 찾아가기 위한 식별자다.

- **IPv4**: `192.168.1.1` (32비트, 약 43억 개)
- **IPv6**: `2001:db8::1` (128비트, 사실상 무한)

IPv4 주소가 고갈되면서 IPv6로 전환이 진행 중이다.

### 공인 IP vs 사설 IP

| 구분 | 설명 | 예시 |
|---|---|---|
| 공인 IP | 인터넷에서 유일한 주소. ISP(통신사)가 부여 | `203.0.113.5` |
| 사설 IP | 내부 네트워크(집, 회사)에서만 쓰는 주소 | `192.168.x.x`, `10.x.x.x` |

집에서 여러 기기가 하나의 인터넷을 쓸 수 있는 건 **NAT(Network Address Translation)** 덕분이다. 공유기가 사설 IP와 공인 IP 사이를 번역해준다.

### 포트 (Port)

하나의 서버에서 여러 서비스를 동시에 운영할 때 포트로 구분한다.

```
203.0.113.5:80    <- 80번 포트 → HTTP 웹 서버
203.0.113.5:443   <- 443번 포트 → HTTPS 웹 서버
203.0.113.5:5432  <- 5432번 포트 → PostgreSQL
203.0.113.5:22    <- 22번 포트 → SSH
```

잘 알려진 포트(Well-known Port)는 0~1023번이며 용도가 정해져 있다. 개발할 때 쓰는 8000, 3000번 등은 등록되지 않은 자유로운 포트다.

### DNS 

사람은 `142.250.196.142` 같은 숫자보다 `google.com` 같은 이름을 기억하기 쉽다. **DNS(Domain Name System)** 는 도메인 이름을 IP 주소로 변환해주는 시스템이다.

```
브라우저에 "google.com" 입력
    │
    V
1. 브라우저 캐시 확인 → 없으면
    │
    V
2. OS 캐시 + /etc/hosts 파일 확인 → 없으면
    │
    V
3. DNS 리졸버(통신사 DNS 서버)에 질의
    │
    V
4. 루트 DNS → .com 담당 DNS → google.com 담당 DNS 순서로 탐색
    │
    V
5. 최종 IP 주소 반환 (142.250.196.142)
    │
    V
6. 브라우저가 해당 IP로 HTTP 요청
```

이 전체 과정이 보통 수십 밀리초 안에 완료된다.

## 브라우저에서 Enter를 누르면 생기는 일

`https://www.google.com`을 입력하고 Enter를 눌렀을 때의 전체 흐름이다. 

```
1. URL 파싱
   브라우저가 프로토콜(https), 도메인(www.google.com), 경로(/) 분리

2. DNS 조회
   www.google.com -> IP 주소 획득 (위 DNS 과정)

3. TCP 연결 (3-Way Handshake)
   클라이언트와 서버가 연결 수립

4. TLS 핸드셰이크 (HTTPS인 경우)
   암호화 통신을 위한 키 교환

5. HTTP 요청 전송
   GET / HTTP/1.1
   Host: www.google.com
   ...

6. 서버 응답
   HTML 문서 반환

7. 브라우저 렌더링
   HTML 파싱 → CSS/JS 다운로드 → 화면에 그림
```

## TCP/IP — 인터넷 통신의 뼈대

인터넷은 **TCP/IP 프로토콜 스택**을 기반으로 동작한다. 4개 계층으로 나뉜다.

```
[ 애플리케이션 계층 ]  HTTP, HTTPS, FTP, SMTP, DNS ...
        ↕                    실제로 주고받는 데이터 형식
[ 전송 계층         ]  TCP, UDP
        ↕                    데이터를 어떻게 전달할지
[ 인터넷 계층       ]  IP
        ↕                    어디로 보낼지 (주소)
[ 네트워크 액세스 계층 ]  이더넷, Wi-Fi
                             실제 전기/광 신호
```

### TCP vs UDP

| 항목 | TCP | UDP |
|---|---|---|
| 연결 방식 | 연결 지향 (3-Way Handshake) | 비연결 |
| 신뢰성 | 순서 보장, 재전송 보장 | 보장 없음 |
| 속도 | 상대적으로 느림 | 빠름 |
| 용도 | HTTP, 파일 전송, 이메일 | 영상 스트리밍, 게임, DNS |

TCP는 데이터가 반드시 도착해야 하는 경우, UDP는 약간의 손실이 있어도 속도가 더 중요한 경우에 쓴다. 유튜브 영상이 중간에 살짝 깨져도 계속 재생되는 이유가 여기에 있다.

### 3-Way Handshake (TCP 연결 수립)

```
클라이언트                          서버
    │──── SYN (연결 요청) ──────────▶│
    │◀─── SYN-ACK (수락 + 요청) ────│
    │──── ACK (확인) ───────────────▶│
    │                                │
    │       (데이터 전송 시작)        │
```

- **SYN**: "나 연결하고 싶어"
- **SYN-ACK**: "알겠어, 나도 연결할게"
- **ACK**: "확인했어"

연결을 끊을 때는 4-Way Handshake를 사용한다.

## HTTP — 웹 통신의 언어

**HTTP(HyperText Transfer Protocol)** 는 클라이언트와 서버가 데이터를 주고받을 때 사용하는 규약이다. 요청(Request)과 응답(Response)의 쌍으로 이루어진다.

### HTTP 요청 구조

```
GET /users/42 HTTP/1.1          ← 요청 라인: 메서드, 경로, 버전
Host: api.example.com           ┐
Authorization: Bearer eyJ...    │ 헤더: 메타 정보
Content-Type: application/json  ┘
                                 ← 빈 줄 (헤더와 본문 구분)
{                               ┐
  "name": "홍길동"               │ 본문(Body): 실제 데이터
}                               ┘
```

### HTTP 메서드

| 메서드 | 의미 | 특징 |
|---|---|---|
| `GET` | 데이터 조회 | Body 없음, 멱등성 O |
| `POST` | 데이터 생성 | Body 있음, 멱등성 X |
| `PUT` | 데이터 전체 교체 | 멱등성 O |
| `PATCH` | 데이터 일부 수정 | 멱등성 O |
| `DELETE` | 데이터 삭제 | 멱등성 O |

> **멱등성(Idempotent)**: 같은 요청을 여러 번 보내도 결과가 동일한 성질. GET /users/1을 100번 해도 결과가 같다. POST는 100번 하면 100개가 생성되므로 멱등하지 않다.

### HTTP 버전 발전

| 버전 | 특징 |
|---|---|
| HTTP/1.0 | 요청마다 TCP 연결을 새로 맺음 → 매우 느림 |
| HTTP/1.1 | **Keep-Alive**: 하나의 TCP 연결로 여러 요청 처리. 현재도 널리 사용 |
| HTTP/2 | **멀티플렉싱**: 하나의 연결에서 여러 요청을 동시에 처리. 헤더 압축 |
| HTTP/3 | TCP 대신 **QUIC(UDP 기반)** 사용. 연결 수립 속도 대폭 향상 |

## HTTPS와 TLS — 암호화 통신

HTTP는 데이터를 **평문(암호화 없이)** 으로 주고받는다. 중간에 누군가 패킷을 가로채면 내용을 그대로 볼 수 있다. **HTTPS**는 HTTP에 **TLS(Transport Layer Security)** 암호화를 더한 것이다.

```
HTTP  → 암호화 없음 → "password=1234" 그대로 노출
HTTPS → TLS 암호화 → "x9f#mK2!..." 해독 불가
```

### TLS 핸드셰이크 (간략 버전)

```
클라이언트                              서버
    │── 안녕, 내가 지원하는 암호화 목록 ──▶│
    │◀─ 알겠어, 이 암호화 방식 쓰자 + 인증서 ─│
    │── 인증서 검증 (CA 신뢰 체인 확인) ──│
    │── 세션 키 협상 ──────────────────▶│
    │                                   │
    │     (이후 대칭키로 암호화 통신)      │
```

**SSL 인증서**는 서버가 신뢰할 수 있는 기관(CA, Certificate Authority)으로부터 발급받은 신분증이다. 브라우저는 이 인증서를 통해 "내가 접속한 서버가 진짜 google.com인가"를 검증한다.

> **HTTPS를 써야 하는 이유**: 암호화뿐만 아니라, 구글이 HTTPS 사이트를 검색 순위에서 우대하고, 최신 브라우저 API(위치 정보, 카메라 등)도 HTTPS에서만 동작한다.

## 상태(State) 관리 — 쿠키, 세션, 토큰

HTTP는 기본적으로 **무상태(Stateless)** 다. 각 요청은 독립적이며, 서버는 이전 요청을 기억하지 못한다. 쇼핑몰에서 로그인한 뒤 다른 페이지로 이동해도 로그인 상태가 유지되는 것은, 무언가가 "이 사람은 로그인한 사람"임을 계속 알려주기 때문이다.

### 쿠키 (Cookie)

브라우저에 저장되는 작은 텍스트 파일이다. 서버가 `Set-Cookie` 헤더로 저장을 지시하면, 이후 같은 서버에 보내는 모든 요청에 브라우저가 자동으로 쿠키를 포함시킨다.

```
서버 응답: Set-Cookie: session_id=abc123; HttpOnly; Secure; Max-Age=3600
이후 요청: Cookie: session_id=abc123   ← 브라우저가 자동 첨부
```

- `HttpOnly`: JavaScript로 쿠키 접근 불가 (XSS 공격 방어)
- `Secure`: HTTPS에서만 전송
- `SameSite`: 다른 도메인에서의 쿠키 전송 제한 (CSRF 방어)

### 세션 기반 인증 (Session-based Auth)

```
① 사용자가 로그인
② 서버: 세션 저장소(메모리/Redis)에 "abc123 = {user_id: 42}" 저장
③ 서버: Set-Cookie: session_id=abc123 응답
④ 이후 요청마다 브라우저가 session_id=abc123 전송
⑤ 서버: session_id로 저장소 조회 → 누군지 확인
```

**장점**: 서버에서 세션을 바로 삭제해 강제 로그아웃 가능  
**단점**: 서버가 세션을 저장해야 하므로 서버가 여러 대이면 세션 공유 문제 발생 (서버 A에 저장된 세션이 서버 B엔 없음)

### 토큰 기반 인증 (JWT)

```
① 사용자가 로그인
② 서버: { user_id: 42, role: "admin" }를 비밀키로 서명 → JWT 생성
③ 서버: JWT를 응답으로 전달 (쿠키 or 응답 바디)
④ 이후 요청: Authorization: Bearer eyJ... 헤더로 JWT 전송
⑤ 서버: JWT 서명 검증 → 유효하면 내용(user_id, role) 신뢰
```

**장점**: 서버가 아무것도 저장하지 않아도 됨 (Stateless). 서버가 몇 대든 같은 비밀키로 검증 가능  
**단점**: 발급된 토큰을 중간에 무효화하기 어려움 (만료 전까지 유효). 토큰 탈취 시 만료까지 사용 가능

> **Refresh Token 패턴**: Access Token은 짧게 유지(15분~1시간)하고, 만료 시 Refresh Token으로 새 Access Token을 발급받는 방식으로 보안과 편의성을 절충한다.

## REST API

**REST(Representational State Transfer)** 는 HTTP를 잘 활용해서 서버와 클라이언트가 통신하는 방식에 대한 설계 원칙이다. 엄격한 표준이 아니라 아키텍처 스타일(가이드라인)이다.

### 핵심 원칙

1. **리소스를 URL로 표현한다** — 동사 대신 명사 사용
   ```
   ❌ /getUser?id=42       /deletePost?id=5
   ✅ /users/42            /posts/5
   ```

2. **행위는 HTTP 메서드로 표현한다**
   ```
   GET    /users       → 전체 유저 목록 조회
   GET    /users/42    → 42번 유저 조회
   POST   /users       → 새 유저 생성
   PUT    /users/42    → 42번 유저 전체 수정
   PATCH  /users/42    → 42번 유저 일부 수정
   DELETE /users/42    → 42번 유저 삭제
   ```

3. **무상태(Stateless)** — 요청에 필요한 정보는 요청 자체에 모두 담는다

4. **계층 구조(Layered System)** — 클라이언트는 중간에 프록시/로드밸런서가 있는지 알 필요 없다

5. **JSON** — 요즘 REST API는 대부분 JSON으로 데이터를 주고받는다

### REST의 한계와 대안

- **과다 응답(Over-fetching)**: 필요한 필드가 2개인데 서버가 100개의 필드를 전부 반환
- **부족한 응답(Under-fetching)**: 화면에 필요한 데이터를 위해 API를 여러 번 호출

이를 해결하기 위해 **GraphQL**이 등장했다. 클라이언트가 필요한 필드만 콕 집어 요청할 수 있다.

## 리버스 프록시 (Reverse Proxy)

**프록시**는 중간에서 대신 요청을 처리하는 서버다.

- **포워드 프록시**: 클라이언트 앞에 위치. 클라이언트를 대신해 요청 (VPN, 기업 내부망)
- **리버스 프록시**: 서버 앞에 위치. 외부 요청을 받아 내부 서버로 전달

```
인터넷
  │
  ▼
[ Nginx — 리버스 프록시 ]
  │  /api/...  →  WAS (FastAPI:8000)
  │  /static/  →  직접 파일 서빙
  │  /ws/...   →  WebSocket 서버
```

**리버스 프록시의 역할**:
- **SSL 종료(SSL Termination)**: HTTPS 처리를 프록시가 맡아 WAS는 HTTP만 처리하면 됨
- **로드 밸런싱**: 여러 WAS 서버로 요청 분산
- **정적 파일 캐싱**: 자주 요청되는 파일을 직접 서빙해 WAS 부하 감소
- **보안**: 내부 서버 IP와 구조를 외부에 노출하지 않음

## 로드 밸런싱 (Load Balancing)

트래픽이 급증하면 서버 한 대로는 감당이 안 된다. **로드 밸런서**는 여러 서버에 요청을 분산시켜 부하를 나눈다.

```
클라이언트들
    │  │  │  │
    ▼  ▼  ▼  ▼
[ 로드 밸런서 ]
  │     │     │
  ▼     ▼     ▼
서버1  서버2  서버3
```

### 분산 알고리즘

| 방식 | 설명 |
|---|---|
| **라운드 로빈** | 순서대로 돌아가며 균등 분배 (기본값) |
| **최소 연결** | 현재 연결 수가 가장 적은 서버로 |
| **IP 해시** | 같은 클라이언트는 항상 같은 서버로 (세션 유지 필요 시) |
| **가중치** | 서버 성능에 따라 비율 조정 |

### 스케일 업 vs 스케일 아웃

| 구분 | 방법 | 특징 |
|---|---|---|
| **스케일 업 (Scale Up)** | 서버 한 대의 CPU/RAM 업그레이드 | 한계가 있고 비용이 큼 |
| **스케일 아웃 (Scale Out)** | 서버 대수를 늘림 | 이론적으로 무한 확장 가능 |

현대 클라우드 아키텍처는 대부분 스케일 아웃을 선택한다. 단, 스케일 아웃을 하려면 서버가 상태를 공유해야 하므로 (세션을 Redis에 저장하는 등) 설계가 복잡해진다.

## 캐싱 (Caching)

**캐시**는 자주 쓰는 데이터를 더 빠른 저장소에 임시로 두어, 매번 원본 소스에서 가져오는 비용을 줄이는 기법이다.

```
요청
 │
 ▼
캐시 히트? ──▶ YES → 캐시에서 바로 응답 (빠름)
     │
     ▼ NO
  원본 DB 조회 → 결과를 캐시에 저장 → 응답 (느림)
```

### 캐싱 레이어

```
[ 브라우저 캐시 ]    ← 가장 가까움. HTTP 헤더로 제어
[ CDN 캐시    ]    ← 전 세계 엣지 서버
[ 리버스 프록시 캐시 ] ← Nginx에서 응답 캐싱
[ 애플리케이션 캐시 ] ← Redis, Memcached
[ DB 쿼리 캐시  ]    ← DB 내부 캐시
```

### HTTP 캐시 헤더

```
Cache-Control: max-age=3600        # 3600초(1시간) 동안 캐시 유효
Cache-Control: no-cache            # 매번 서버에 재검증 요청
Cache-Control: no-store            # 캐시 저장 자체를 금지
ETag: "abc123"                     # 콘텐츠 해시값. 변경 여부 확인용
```

### 캐시 무효화 (Cache Invalidation)

캐시의 가장 어려운 문제는 **언제 버릴지** 결정하는 것이다. 원본 데이터가 바뀌었는데 캐시가 오래된 데이터를 계속 반환하면 문제가 생긴다.

## CDN (Content Delivery Network)

**CDN**은 전 세계 여러 곳에 서버(엣지 서버)를 두고, 사용자와 **가장 가까운 서버**에서 콘텐츠를 제공하는 시스템이다.

```
한국 사용자 → 서울 엣지 서버에서 응답 (20ms)
미국 사용자 → 뉴욕 엣지 서버에서 응답 (15ms)
(원본 서버가 서울에 있어도, 미국 사용자는 뉴욕에서 받음)
```

정적 파일(이미지, JS, CSS), 영상 스트리밍처럼 자주 바뀌지 않는 대용량 파일에 특히 효과적이다.

## WebSocket — 실시간 양방향 통신

HTTP는 클라이언트가 요청해야만 서버가 응답한다. 채팅처럼 서버가 먼저 데이터를 밀어줘야 할 때는 **WebSocket**을 쓴다.

```
[ HTTP ]
클라이언트 ──── 요청 ────▶ 서버
            ◀─── 응답 ────

[ WebSocket ]
클라이언트 ──── Upgrade 요청 ───▶ 서버
            ◀── 101 Switching ────
            (이후 TCP 연결 유지)
클라이언트 ◀───────────────────── 서버 (서버가 먼저 데이터 전송 가능)
클라이언트 ──────────────────────▶ 서버 (클라이언트도 자유롭게 전송)
```

WebSocket은 HTTP 업그레이드 요청으로 시작해 TCP 연결을 계속 유지한다. 채팅, 실시간 알림, 주식 차트, 온라인 게임 등에 사용된다.

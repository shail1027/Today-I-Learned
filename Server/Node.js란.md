## Node.js란?

### Node.js란?

Node.js는 Chrome V8 JavaScript 엔진으로 빌드된 **JavaScript 런타임(Runtime)** 이다.
과거에는 자바스크립트가 웹 브라우저 안에서만 동작하는 프론트엔드 전용 언어였으나, Node.js의 등장으로 브라우저 밖인 서버(백엔드) 환경이나 OS 시스템 위에서도 자바스크립트를 실행할 수 있게 되었다. 즉, 웹 서버를 구축하거나 운영체제 시스템 스크립트를 작성하는 등 자바스크립트의 활용 범위를 무한히 넓혀준 실행 환경이라고 볼 수 있다.

-----

### 비동기 I/O (Non-blocking I/O)

Node.js의 가장 큰 성능적 특징은 **Non-blocking I/O**를 사용한다는 점이다.

  * 시스템에서 파일 읽기/쓰기, 네트워크 요청, 데이터베이스 쿼리 등은 CPU 연산에 비해 속도가 매우 느린 I/O 작업이다.
  * 기존의 동기(Blocking) 방식은 이러한 I/O 작업이 끝날 때까지 스레드가 아무것도 하지 않고 대기해야 했다.
  * 하지만 Node.js는 I/O 작업을 백그라운드(시스템 OS 등)로 넘기고, 곧바로 다음 코드를 실행한다. 이후 백그라운드 작업이 완료되면 그때 결과를 받아(콜백 등) 나머지 처리를 진행하므로 자원의 낭비가 없다.

-----

### 이벤트 기반 (Event-Driven)과 이벤트 루프 (Event Loop)

Node.js는 이벤트가 발생할 때 미리 지정해둔 작업(콜백 함수)을 수행하는 **이벤트 기반** 방식으로 동작한다.
이 중심에는 **이벤트 루프(Event Loop)** 가 있다. 이벤트 루프는 여러 큐(Queue)와 스택(Stack)을 관리하며, 비동기 작업이 완료되어 콜백 함수가 큐에 쌓이면 이를 차례대로 메인 스레드(Call Stack)로 가져와 실행하는 핵심 엔진이다.

-----

### 싱글 스레드 (Single Thread)의 의미와 한계

소프트웨어 구조 관점에서 Node.js는 흔히 **싱글 스레드**로 동작한다고 표현한다. 정확히는 자바스크립트 코드를 실행하는 '메인 스레드(이벤트 루프)'가 단 하나라는 뜻이다.

  * **장점:** 스레드를 여러 개 생성하고 관리하는 오버헤드나, 스레드 간의 자원 공유 문제(Deadlock 등)를 신경 쓸 필요가 없어 구조가 단순하고 메모리 효율이 좋다. 단 하나의 스레드가 Non-blocking I/O를 통해 수만 개의 동시 접속을 가볍게 처리할 수 있다.
  * **단점 (한계):** 메인 스레드가 단 하나이므로, 이미지 압축이나 복잡한 수학 계산과 같이 **CPU 연산이 오래 걸리는 작업(CPU-intensive task)**을 만나면 해당 연산이 끝날 때까지 다른 모든 요청의 처리가 중단되는 병목(Blocking) 현상이 발생한다. 따라서 CPU 집약적인 서버보다는 I/O 집약적인 서버에 적합하다.

-----

### 패키지 매니저: npm (Node Package Manager)

Node.js를 설치하면 `npm`이 함께 설치된다. 이는 전 세계의 개발자들이 만들어 놓은 수많은 자바스크립트 라이브러리(패키지)를 쉽게 설치하고 관리할 수 있게 해주는 도구이다. 프론트엔드의 React, Vue부터 백엔드의 Express, NestJS 등 생태계의 모든 것이 이 npm을 통해 이루어진다.

-----

### 언제 사용해야 할까?

  * **적합한 경우:** 채팅 어플리케이션, 주식 차트 등 **실시간(Real-time)** 양방향 통신이 필요한 서비스.
      * 넷플릭스와 같은 데이터 스트리밍 서비스.
      * JSON 기반의 빠르고 가벼운 REST API 서버.
      * 프론트엔드와 백엔드를 동일한 언어(JavaScript/TypeScript)로 개발하여 팀의 생산성을 높이고 싶을 때.
  * **부적합한 경우:**
      * 대규모 비디오 인코딩, 복잡한 머신러닝 연산 등 CPU 사용량이 극도로 높은 서비스.

# 프로세스와 스레드

## 프로그램 vs 프로세스 vs 스레드

세 개념은 자주 혼용되지만 엄연히 다르다.

| 개념 | 설명 | 비유 |
|---|---|---|
| **프로그램(Program)** | 디스크에 저장된 실행 파일. 아직 실행되지 않은 정적인 코드 덩어리 | 악보 |
| **프로세스(Process)** | 프로그램이 메모리에 올라와 실행 중인 상태. OS가 자원을 할당한 독립 단위 | 악보를 연주하는 오케스트라 |
| **스레드(Thread)** | 프로세스 안에서 실행되는 작업 단위. 프로세스의 자원을 공유함 | 오케스트라의 각 악기 파트 |

---

## 프로세스의 구조

프로세스가 생성되면 OS는 메모리를 4개 영역으로 나눠 할당한다.

```
높은 주소
┌─────────────┐
│    Stack    │  ← 함수 호출 정보, 지역 변수 (LIFO, 위에서 아래로 쌓임)
│      ↓      │
│             │
│      ↑      │
│    Heap     │  ← 동적 할당 메모리 (malloc, new 등, 아래에서 위로)
├─────────────┤
│    Data     │  ← 전역 변수, static 변수 (프로그램 시작 시 할당)
├─────────────┤
│    Code     │  ← 실행할 기계어 코드 (읽기 전용)
└─────────────┘
낮은 주소
```

- **Code(Text)**: `if`, `for`, 함수 정의 등 실제 명령어가 들어 있다. 읽기 전용이라 실행 중에 변경되지 않는다.
- **Data**: 전역 변수(`int count = 0;`)나 `static` 변수가 저장된다. 프로그램 종료 때까지 유지된다.
- **Heap**: 런타임에 동적으로 할당되는 메모리다. 직접 관리해야 하며, 해제하지 않으면 메모리 누수가 발생한다.
- **Stack**: 함수가 호출될 때마다 그 함수의 지역 변수, 매개변수, 반환 주소가 쌓인다. 함수가 종료되면 자동으로 해제된다.

### PCB — 프로세스 제어 블록

OS는 각 프로세스를 관리하기 위해 **PCB(Process Control Block)** 라는 자료구조를 유지한다. 여기에 프로세스의 모든 상태 정보가 담긴다.

```
PCB
├── PID (프로세스 고유 ID)
├── 프로세스 상태 (실행 중, 대기 중, 종료 등)
├── PC (Program Counter) — 다음 실행할 명령어 주소
├── CPU 레지스터 값들
├── 메모리 관련 정보 (코드/스택/힙 주소)
├── 열린 파일 목록
└── 우선순위 등
```

---

## 프로세스의 상태

```
                  ┌─────────────────────────────┐
     생성          │            대기              │
      │            │     (I/O 완료 등 이벤트 대기) │
      ▼            └──────────────┬──────────────┘
  [ 준비 ]  ◀── 타임슬라이스 만료 ──┘
      │                           ▲
      │  CPU 할당                 │ I/O 요청 등
      ▼                           │
  [ 실행 ] ────────────────────▶ [ 대기 ]
      │
      │ 작업 완료
      ▼
  [ 종료 ]
```

- **준비(Ready)**: CPU를 기다리는 상태. 메모리엔 올라와 있지만 아직 실행 안 됨
- **실행(Running)**: CPU를 점유하며 명령어를 실행 중
- **대기(Waiting/Blocked)**: I/O 작업 등 특정 이벤트를 기다리는 상태. CPU를 쓰지 않음
- **종료(Terminated)**: 실행이 끝난 상태

---

## 스레드의 구조

스레드는 프로세스 안에서 실행되는 작업 흐름이다. **같은 프로세스의 스레드들은 대부분의 자원을 공유하고, Stack만 독립적으로 가진다.**

```
┌──────────────────────────────────────────┐
│                 프로세스                  │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ Thread 1 │  │ Thread 2 │  │Thread 3│ │
│  │  Stack   │  │  Stack   │  │ Stack  │ │  ← 각자 독립
│  │    PC    │  │    PC    │  │   PC   │ │  ← 각자 독립
│  └──────────┘  └──────────┘  └────────┘ │
│                                          │
│  ┌──────────────────────────────────┐   │
│  │    Code + Data + Heap + 파일     │   │  ← 전부 공유
│  └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
```

스레드가 Stack을 독립적으로 갖는 이유는, 각 스레드가 **각자 다른 함수를 실행**하고 있기 때문이다. 함수 호출 흐름이 다르면 스택도 달라야 한다.

---

## 프로세스 vs 스레드 비교

| 항목 | 프로세스 | 스레드 |
|---|---|---|
| 메모리 | 독립적인 메모리 공간 | Code, Data, Heap 공유 |
| 생성 비용 | 크다 (메모리 복사 등) | 작다 (Stack만 추가) |
| 통신 방법 | IPC (파이프, 소켓, 공유 메모리) | 전역 변수, Heap 직접 접근 |
| 안전성 | 한 프로세스 죽어도 다른 프로세스에 영향 없음 | 한 스레드 오류가 프로세스 전체에 영향 |
| 컨텍스트 스위칭 비용 | 크다 | 작다 |

## 동시성 vs 병렬성

자주 혼용되지만 핵심이 다른 개념이다.

### 동시성 (Concurrency)

**여러 작업을 번갈아가며** 처리해 동시에 진행되는 것처럼 보이게 하는 것. CPU가 1개여도 가능하다.

```
CPU 1개

시간 →  ─────────────────────────────────▶
        [Task A][Task B][Task A][Task B][Task A]
         빠르게 번갈아 실행 → 동시처럼 보임
```

### 병렬성 (Parallelism)

**여러 작업을 물리적으로 동시에** 처리하는 것. CPU 코어가 여러 개여야 가능하다.

```
CPU 4개

Core 1: [Task A ──────────────────]
Core 2: [Task B ──────────────────]
Core 3: [Task C ──────────────────]
Core 4: [Task D ──────────────────]
         실제로 동시에 실행
```

> "동시성은 여러 일을 다루는 것(dealing with), 병렬성은 여러 일을 실제로 하는 것(doing)"

---

## 컨텍스트 스위칭 (Context Switching)

CPU가 실행 중인 프로세스(또는 스레드)를 바꿀 때 일어나는 작업이다.

```
프로세스 A 실행 중
    │
    │ 타임슬라이스 만료 or I/O 발생
    ▼
OS 개입
    ├── 현재 상태(PC, 레지스터, ...) → PCB_A에 저장
    └── PCB_B에서 상태 복원 → 프로세스 B 실행
```

컨텍스트 스위칭은 CPU가 실제 작업을 하지 않는 **순수한 오버헤드**다. 스레드 간 전환이 프로세스 간 전환보다 빠른 이유는, 메모리 공간(페이지 테이블 등)을 교체할 필요가 없기 때문이다.

## 멀티프로세싱 vs 멀티스레딩

### 멀티프로세싱 (Multi-processing)

여러 프로세스를 동시에 실행한다. 각 프로세스가 완전히 독립된 메모리를 사용한다.

```
부모 프로세스 (PID 100)
    │
    ├── 자식 프로세스 A (PID 101) — 독립 메모리
    ├── 자식 프로세스 B (PID 102) — 독립 메모리
    └── 자식 프로세스 C (PID 103) — 독립 메모리
```

**장점**: 한 프로세스가 죽어도 나머지에 영향 없음. GIL 회피 가능 (Python에서 중요)  
**단점**: 생성 비용이 크고, 프로세스 간 통신(IPC)이 복잡하고 느림

### 멀티스레딩 (Multi-threading)

하나의 프로세스 안에서 여러 스레드가 동시에 실행된다.

```
프로세스 (PID 100)
    │
    ├── Thread 1 — 공유 메모리 접근
    ├── Thread 2 — 공유 메모리 접근
    └── Thread 3 — 공유 메모리 접근
```

**장점**: 생성 비용이 작고, 스레드 간 데이터 공유가 쉬움  
**단점**: 공유 자원 접근 시 동기화 필요. 한 스레드의 오류가 전체에 영향 가능

---

## 동기화 문제

여러 스레드가 같은 자원에 동시에 접근하면 문제가 생긴다.

### 경쟁 상태 (Race Condition)

두 스레드가 같은 변수를 동시에 읽고 쓸 때, 실행 순서에 따라 결과가 달라지는 현상이다.

```python
# 전역 변수 count = 0, 스레드 2개가 각각 1000번 더할 때

# 기대 결과: 2000
# 실제 결과: 1342, 1756, 1923... (실행할 때마다 다름)

# 이유: "count += 1" 은 실제로 3단계
# 1. count 읽기
# 2. +1 계산
# 3. count에 쓰기
# → 두 스레드가 동시에 1단계를 하면 서로의 쓰기가 덮어씌워짐
```

### 임계 구역 (Critical Section)

Race Condition이 발생할 수 있는 코드 구간이다. 한 번에 하나의 스레드만 진입해야 한다.

### Mutex (뮤텍스)

**상호 배제(Mutual Exclusion)** 를 구현하는 잠금이다. 하나의 스레드가 잠금을 획득하면, 나머지 스레드는 잠금이 풀릴 때까지 대기한다.

```
Thread A        Mutex       Thread B
   │                            │
   ├── lock() ──▶ 잠금           │
   │           (Thread B 대기)  │
   │             ◀──────────── lock() 시도
   │   임계 구역 실행             │
   ├── unlock() ▶ 잠금 해제      │
                  ──────────▶ lock() 획득
                               임계 구역 실행
                               unlock()
```

### Semaphore (세마포어)

정수 카운터로 **동시에 접근 가능한 스레드 수**를 제한한다. Mutex가 "1명만 들어가"라면, Semaphore는 "최대 N명까지 들어가"다.

```
Semaphore(3) → 동시에 3개 스레드까지 허용

DB Connection Pool이 3개라면:
→ 4번째 요청은 연결이 반납될 때까지 대기
```

### 교착 상태 (Deadlock)

두 스레드가 서로 상대방의 잠금이 풀리기를 기다리며 **영원히 멈추는** 상태다.

```
Thread A: Lock(자원1) 보유 중, Lock(자원2) 대기
Thread B: Lock(자원2) 보유 중, Lock(자원1) 대기
→ 둘 다 영원히 기다림
```

**발생 조건 4가지** (하나라도 깨면 Deadlock 방지):
1. 상호 배제 — 자원을 하나의 스레드만 사용 가능
2. 점유와 대기 — 자원을 보유하면서 다른 자원을 기다림
3. 비선점 — 자원을 강제로 빼앗을 수 없음
4. 순환 대기 — A→B→C→A 식으로 대기가 순환

## Python의 GIL

Python(CPython)에는 **GIL(Global Interpreter Lock)** 이라는 특수한 잠금이 있다. 한 번에 하나의 스레드만 Python 코드를 실행할 수 있게 막는 뮤텍스다.

```
GIL 때문에 Python 멀티스레딩의 실제 모습

기대:
Core 1: [Thread A ────────]
Core 2: [Thread B ────────]

실제:
Core 1: [Thread A][Thread B][Thread A][Thread B]
Core 2: (놀고 있음)
→ CPU 연산은 사실상 단일 코어만 사용
```

**왜 GIL이 존재하는가**: CPython의 메모리 관리(참조 카운팅)가 스레드 안전하지 않아, 단순하고 빠른 해결책으로 도입됐다.

**GIL의 영향**:
- **I/O 바운드 작업** (파일, 네트워크): I/O 대기 중에는 GIL을 놓아주므로 멀티스레딩이 효과 있음
- **CPU 바운드 작업** (수학 연산, 이미지 처리): GIL 때문에 코어를 여러 개 써도 빨라지지 않음

**해결책**:
```
CPU 바운드  →  multiprocessing 모듈 사용 (각 프로세스는 GIL 독립)
I/O 바운드  →  threading 또는 asyncio 사용
```

## 언제 무엇을 선택할까?

```
작업의 성격
    │
    ├── I/O 대기가 많다 (네트워크, 파일, DB)
    │       │
    │       ├── 단순하고 빠른 처리
    │       │   → asyncio (코루틴, 단일 스레드)
    │       │
    │       └── 라이브러리가 비동기를 지원하지 않음
    │           → threading (멀티스레딩)
    │
    └── CPU 연산이 많다 (수학 계산, 이미지/영상 처리, ML)
            │
            └── multiprocessing (멀티프로세싱)
```

| 방식 | 병렬성 | 메모리 | 적합한 작업 |
|---|---|---|---|
| `threading` | 동시성 (GIL로 제한) | 공유 | I/O 바운드 |
| `multiprocessing` | 진정한 병렬성 | 독립 | CPU 바운드 |
| `asyncio` | 동시성 (단일 스레드) | 공유 | I/O 바운드 + 대량 연결 |

## 정리

```
프로그램  →  실행 전 파일
프로세스  →  실행 중인 프로그램 (독립 메모리)
스레드    →  프로세스 내 작업 단위 (메모리 공유)

동시성  →  번갈아 실행 (CPU 1개로 가능)
병렬성  →  동시에 실행 (CPU 여러 개 필요)

공유 자원 접근 시 주의
├── Race Condition  →  실행 순서에 따라 결과가 달라짐
├── Mutex           →  한 번에 1개 스레드만 임계 구역 진입
├── Semaphore       →  N개 스레드까지 동시 접근 허용
└── Deadlock        →  서로 기다리며 영원히 멈춤
```
